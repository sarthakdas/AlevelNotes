Software Development

There are infinite variety of ways a problem an be solved using a computer. Stages of analysis, design, implementation, installation and evaluation are common to all projects.

Analysis

A problem must first be defined. This should cover:

  • Data - its origin, uses, volumes and characteristics
  • Procedures - what is done where, when, and how, and how errors are handled
  • Problems with any existing system

Design

Depending on the type of project they might have to consider the following:

  • Processing - the algorithms and appropriate modular structure
  • Data structures - how data will be held and accessed
  • Output - content, format, sequence, frequency, medium
  • Input - volume, frequency, documents used, input metheords
  • User Interface - screens and dialogues, menus
  • Security - how data is kept secure
  • Hardware - selection of an appropriate configuration

Testing

This generally involves breaking a problem into individual components until each of these modules preform a single well-defined task. These are several testing strategies:

Black Box Testing (function testing)

It is carries out independently from the code. It involves looking at the program specification and creating a set of test data that covers all the inputs and outputs and program functions.

White Box Testing (structural testing)

It is dependent of code logic, and derives from the program structure rather than its function. The program code is studied and tests are devised which test each possible path at least once.

Alpha Testing

It is carried out by the software developer's in-house testing team. It is essential because it is often reveals both errors and omissions in the system requirements. This can solve problems in terms of overlooking and misunderstanding the specification.

Beta Testing

This involves giving the package to a number of potential users who agree to use the system and report any problem to the developers. This exposes the product to real use and detects problems and errors that may not have been anticipated.

Implementation

Coding and testing will be carried out. When it is all satisfactory the software will be installed on the user's system and more testing will be done.

Evaluation

This may include a post-implementation review, which is a critical examination of the system three to six months after it has been put into operation. This gives waiting time for training and adjustments, and the management a chance to evaluate the usefulness of the reports. So eventually all shortcomings will be apparent through all levels of the organisation.

It will focus on:

  • Comparison of actual and anticipated performance
  • Errors which have been made during development
  • Unexpected benefits and problems

Types of Modelling and Methodology

Waterfall Lifecycle Model

This is when each step is completed one at a time from beginning to end. This was adopted from the manufacturing industry but had high implications if work was done wrong.

Spiral Model

This method still uses the same steps but introduces the idea of developing in iterative repeating steps. Generally used for large scale projects. Each time around the spiral the following are performed:

  • Analyse the requirements for the next prototype
  • Design the next version
  • Implement the new prototype
  • Evaluate the new prototype

Agile Modelling

This is an iterative process which changes made are incremental as the next part of the system are built. At each stage, a prototype is built with user participation. The sucess of he development depends on:

  • Keeping the model simple
  • Rapid user feedback
  • Understanding user requirements might change
  • Incremental changes

Exterme Programming

It is a methodology which is intended to improve software quality and responsiveness to changing customer requirements. It is a type of agile development where frequent updates are made. It improves productivity and introduces checkpoints at which new requirements can be added.

Rapid Application Development

Very large projects during which technology and user requirements might change. Major changes at late stages can lead to projects being cancelled or being restated. RAD methodology offers faster completion of major projects. The ideas behind this include:

  • Workshops and focus groups
  • Use of prototyping to continually refine the system
  • Producing within a strict time limit each part of the system, which may not be perfect but good enough.
  • Reusing software components

Relative Merits and Drawbacks of each Methodology

  • The Waterfall Model is suitable for very small projects which need careful supervision. Absence of user feedback is a drawback
  • The Spiral Model and Agile approach are acknowledge that requirements might change. It is easier to to examine a working prototype and figure out what needs to be done.
  • Extreme Programming and Rapid Application Development are good for large projects where there is a chance of getting sidetracked by suggested improvements, so developers are continually chasing a moving target.

Writing and Following Algorithms

A good algorithm has the following properties:

  • Clear and precise steps
  • Allows for invalid inputs
  • Terminates at one point
  • Executes efficiently
  • Other people should be able to read and understand it

Problems solved by Algorithms

  • Internet-related used to manipulate huge amount of data on the internet. Search Engines
  • Route-finding given two locations, this is used to determine the shortest route between the two points.
  • Compression used to compress data files so they can be transmitted faster or be held in smaller space.
  • Encryption used to hide data when being transmitted to avoid others from reading intercepted messages.
